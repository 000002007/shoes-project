# Шаг 1 — Модель → атрибуты (дизайн)

**Дата:** 2026-06-15
**Статус:** согласовано, ожидает ревью перед планом реализации
**Трекер:** [ROADMAP](../../ROADMAP.md)

## Цель и критерий готовности

Автономный кусок: пользователь вводит название модели кроссовок → система
возвращает **структурированные атрибуты** с уровнем уверенности → пользователь
может поправить любое поле и подтвердить.

Стопы и вердикта здесь нет — это шаги 2–3.

**Готово, когда:** по названию модели система отдаёт нормализованный объект
атрибутов с меткой уверенности, валидирует его по схеме, и пользователь может
подтвердить/исправить результат.

## Ограничения (зафиксированы)

- Стек: Vite + React + Node-сервер.
- LLM сейчас — заглушка (без затрат); потом Claude API + веб-поиск (вариант b),
  переключение через env.
- Ключ API только на сервере, не во фронте.
- Против галлюцинаций: метка уверенности + подтверждение пользователем (базы нет).

## Архитектурный подход: интерфейс провайдера

Один интерфейс «дай атрибуты по названию» с двумя сменными реализациями,
выбор через переменную окружения:

- **MockProvider** — сейчас. Готовые фикстуры на 5–10 моделей. Ноль затрат, легко тестировать.
- **AnthropicProvider** — потом. Claude API; веб-поиск (вариант b) — настройка внутри.
  Сейчас это заглушка, возвращающая «не настроено».

Почему так: всю цепочку шага 1 можно построить и протестировать сейчас без ключа
и без сети; подключение реального LLM — это смена env, без переписывания.

## Поля атрибутов

| Поле | Тип / значения | Примечание |
| --- | --- | --- |
| `brand` | строка | бренд |
| `model` | строка | каноническое имя модели |
| `category` | running / lifestyle / basketball / training / hiking / other | |
| `upperMaterial` | строка | напр. «engineered mesh», «кожа» |
| `stretch` | stretchy / moderate / rigid | тянется ли верх |
| `sizeReputation` | runs_small / true_to_size / runs_large | как модель в размер |
| `widthReputation` | narrow / standard / wide | репутация колодки |
| `toeBox` | low / standard / roomy | необязательное |
| `confidence` | high / medium / low | общая уверенность |
| `notes` | строка | короткая заметка о посадке (необязательно) |
| `sources` | список ссылок | для варианта b; у заглушки пусто |

**Правило честности:** при низкой уверенности поле = `unknown`. Никаких выдуманных
миллиметров — только категории.

## Компоненты

**Сервер**
- `provider` — интерфейс + фабрика (mock / anthropic по env).
- `mockProvider` — фикстуры на 5–10 популярных моделей.
- `anthropicProvider` — сейчас заглушка «не настроено»; потом Claude API + веб-поиск.
- `attributes`-схема + валидация (zod).
- Эндпоинт `POST /api/lookup`.

**Фронт**
- Поле ввода модели.
- Карточка результата с бейджем уверенности.
- Редактируемые поля атрибутов.
- Кнопка «Подтвердить».

## Поток данных

1. Пользователь вводит название → `POST /api/lookup { model }`.
2. Сервер нормализует имя → `provider.lookup()` → объект атрибутов.
3. Валидация по схеме (zod).
4. Фронт рисует карточку → пользователь правит/подтверждает.
5. Подтверждённый результат пока живёт в состоянии приложения (БД — шаг 5).

## Обработка ошибок

- Нет ключа / ошибка провайдера → дружелюбное сообщение + ручной ввод атрибутов.
- Низкая уверенность → подсветка «проверь».
- Неизвестная модель → `confidence: low` + пустые поля, просим заполнить руками.

## Тестирование (TDD)

- Валидация схемы атрибутов.
- MockProvider возвращает валидную форму.
- Нормализация имени модели.
- Эндпоинт `POST /api/lookup` отдаёт корректную структуру.

Всё тестируется без сети и без ключа API.
