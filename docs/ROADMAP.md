# shoes-project — Роадмап и трекер workflow

Веб-приложение для подбора кроссовок по удобству. Этот файл — **единый трекер
прогресса**: фазы, зафиксированные решения и статусы. Обновляем статусы по ходу.

> **Как пользоваться:** у каждой фазы есть статус. Меняем по мере работы.
> Статусы: `⬜ не начато` · `🟡 в работе` · `✅ готово` · `⏸ отложено (потом)`.

---

## Честное обещание продукта

Подсказка о посадке кроссовок на основе замеров стопы и репутации модели.
**Это ориентир, а не примерка.** Точного замера комфорта не обещаем — данные
колодок производителей закрыты. Сильная сторона — **сравнение относительно
обуви, которую пользователь уже удобно носит**.

Дисклеймер на каждом вердикте обязателен.

---

## Зафиксированные решения

| Тема | Решение |
| --- | --- |
| Стек | Vite + React + Node-сервер (как в wc26-predictor) |
| LLM сейчас | Заглушка (mock), без затрат |
| LLM потом | Claude API + веб-поиск (вариант b), переключение через env |
| Ключ API | На сервере, не во фронте. Пока ключа нет — работает заглушка |
| Вердикт | Под строгими правилами; LLM только формулирует объяснение |
| Замер стопы | Ручной (A4) + опция прямого ввода; размер опционален |
| Антигаллюцинации | Метка уверенности + подтверждение пользователем (вместо базы) |
| База данных | Отложена (шаг 5): сначала кеш, потом каталог для рекомендаций |

## Фазы

### ⬜ Шаг 0 — Что обещаем пользователю
Зафиксировать честное обещание и дисклеймер. _(см. раздел выше — по сути готово,
подтвердить формулировку при первом UI)._
- **Готово, когда:** обещание сформулировано одним предложением и мы способны его
  сдержать честно.

### 🟡 Шаг 1 — Модель → атрибуты (LLM-скелет, без базы)
Ввёл название модели → структурированные атрибуты + уверенность → правка/подтверждение.
Спека: [step1-model-attributes-design](superpowers/specs/2026-06-15-step1-model-attributes-design.md).
- **Входит в MVP.** **Без базы.** LLM — заглушка, потом Claude API + веб-поиск.
- **Готово, когда:** по названию модели система отдаёт нормализованный объект
  атрибутов с меткой уверенности, и пользователь может подтвердить/поправить.

### ⬜ Шаг 2 — Ручной замер стопы
Ввод длины (A4) / ориентир по удобной модели; ширина и размер — опционально, с
предупреждением о предвзятости при неполных данных.
- **Входит в MVP.**
- **Готово, когда:** система хранит замер нормализованно (мм + категория ширины) и
  переводит в размеры разных систем.

### ⬜ Шаг 3 — Вердикт о комфорте (подойдёт / тесно / свободно)
Детерминированные **правила** дают вердикт; LLM пишет человеческое объяснение.
- **Входит в MVP.**
- **Готово, когда:** на вход (стопа + модель) выдаётся стабильный вердикт из 3
  категорий + уверенность + причина + дисклеймер.

### ⏸ Шаг 4 — Автозамер стопы по фото
Высокая сложность для одиночки. Откладываем; обходной путь — ручной A4 (шаг 2).

### ⏸ Шаг 5 — База данных
Сначала кеш проверенных моделей, затем каталог для **рекомендаций** брендовых
кроссовок. Появляется после того, как вердикт докажет ценность.

---

## Граница MVP

**Обязательно (и больше ничего):** цепочка `модель → атрибуты → стопа → вердикт`
(шаги 1–3). Если функция не в этой цепочке — она не в MVP.

**Потом:** фото-замер (4), база/каталог рекомендаций (5), аккаунты/история,
учёт формы стопы (подъём, косточки), монетизация.

---

## Честность оценки комфорта

Данных колодок нет → даём **грубую категориальную** оценку (подойдёт/тесно/свободно)
по: длина стопы vs номинал размера + запас в носке; ширина стопы vs репутация
колодки; тянется ли материал; репутация по размеру. Никаких выдуманных
миллиметров — только категории и уровень уверенности. Чем меньше данных дал
пользователь — тем ниже показанная уверенность.

## Стек

- **Фронт+бэк:** Vite + React (фронт) + Node-сервер (прячет ключ LLM).
- **LLM:** интерфейс провайдера; заглушка сейчас, Claude API + веб-поиск потом.
- **Хранилище:** на MVP не нужно; кеш/БД — шаг 5.
- **Вердикт:** обычный код (правила), LLM — только текст объяснения.
