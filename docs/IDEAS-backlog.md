# Бэклог идей улучшений

Сюда складываем согласованные идеи, чтобы не потерялись и можно было «если что добавить».
Источник — обсуждение 2026-06-16. Статусы: `⬜ не начато` · `🟡 в работе` · `✅ сделано` · `⏸ отложено`.

## Принцип-фильтр (важно)

Новые данные о стопе/обуви берём **только** там, где есть честный источник
(известная фактура **или** ввод пользователя) **И** где они реально меняют вердикт
обоснованно. Данные «для красоты», которые ничего честно не уточняют, — это лишняя
поверхность для галлюцинаций. Через этот фильтр и отобраны пункты ниже.

## Приоритет 1 — быстрый выигрыш

### ⬜ Позитивные инструкции замера (UX)
Подписи у полей замера сейчас говорят «как НЕ надо» («это НЕ длина стельки и НЕ размер»).
Переписать на «как НАДО»:
> Как измерить: встань на лист бумаги, обведи стопу карандашом вертикально, измерь
> линейкой от пятки до самого длинного пальца (в см). Мерь вечером и обе стопы — бери большую.
- **Честность/ценность:** меньше ошибок ввода → честнее результат. Усилия крошечные.
- **Где:** `src/components/FootInput.tsx` (подписи), готово к применению хоть сейчас.

## Приоритет 2 — вместе с шагом 4 (фидбэк словами)

### ⬜ Материал → растяжимость / разнашивание (обосновано исследованием)
Грунт: [research/foot-and-shoe-notes.md §3.1](research/foot-and-shoe-notes.md). Кожа тянется
«аномально» (коэф. Пуассона до 1.8) → реально **разнашивается и облегает**; трикотаж (Flyknit) —
**зональная** растяжимость; синтетика/mesh — жёсткая, не даёт; подкладка снижает жёсткость.
- **Два ОРТОГОНАЛЬНЫХ сигнала** (ключевой вывод, не смешивать):
  - `instantGive` — мгновенная податливость из `attrs.stretch` (stretchy > moderate > rigid).
  - `breaksIn` — разнашивание из **семейства материала** (`src/fit/material.ts`): кожа/замша = да;
    трикотаж/синтетика/unknown = нет. Ловушки фикстур: кожа бывает `rigid`, но разнашивается;
    Primeknit `stretchy`, но не разнашивается навсегда.
- **Честность/ценность:** снижает галлюцинации — берём из правил, а не из догадки LLM. Разнашивание
  привязано СТРОГО к коже; синтетике — честный контр-сигнал «жёсткий, теснота останется»; unknown → молчим.
- **Где:** шаг 4 (вердикт) — детальные правила уже в ROADMAP §«Шаг 4». Усилия низкие-средние.
- **Вне fit-вердикта:** подошва/межподошва (EVA/TPU, сцепление/амортизация — §3.2/3.3) — это про
  ход, не про посадку. Не подмешивать.

## Приоритет 3 — отдельная фича, высокая ценность

### ⬜ Сравнение с обувью, которую уже удобно носишь
Пользователь указывает модель/размер обуви, которую комфортно носит → якорь на **реальные
данные** для калибровки вердикта («ты удобно носишь X EU 42, эта модель малит → бери 43»).
- **Честность/ценность:** высшая честность — реальный якорь против закрытых колодок.
  Прямо заявлено сильной стороной продукта. Самый сильный рычаг из всех идей.
- **🔑 Единственный честный путь к обхвату/объёму:** по исследованию (§2) посадку решают обхват
  (ball girth) и объём — их из закрытых колодок не взять. Удобная обувь — это **пользовательская
  «правда»**, неявно кодирующая обхват/объём для своего контура → превращает наше популяционное
  приближение в **калибровку под человека**, ничего не выдумывая.
- **⚠️ Граница:** калибрует только **длину/ширину**; перенести обхват/подъём/форму колодки между
  двумя РАЗНЫМИ моделями нельзя. Двухпанельное сравнение (одна стопа, два контура) — честная
  примерка; «ранжировать похожие модели» = сознательный сдвиг к **подбору** (нужен ОК владельца).
- **Где:** отдельная спека (хранение «эталонной» обуви + сопоставление). Усилия средние.

### ⬜ Шаблон посадки («рентген-отпечаток» удобной обуви) + поиск похожего  — развитие П3
Пользователь берёт модель+размер обуви, которую уже удобно носит (подтягиваем атрибуты
тем же lookup'ом), смотрит её как **рентген-оверлей** (движок шага 3) и **сохраняет это как
шаблон** — «отпечаток» того, как нога сидит в комфортной обуви. Дальше — **искать похожее**:
другие модели, чьи внутренние контуры близки к шаблону.
- **Честность/ценность:** высший якорь — реальная «эта обувь мне удобна». Хитрость: комфортная
  обувь **калибрует наше приближение под конкретного человека** (ошибка по закрытой колодке
  частично гасится — у нас есть реальная точка «здесь впору»). Переиспользует движок оверлея шага 3.
- **⚠️ Сдвиг позиционирования:** «искать похожее» — это уже **подбор/рекомендация**, а сейчас
  продукт строго **примерка** (CLAUDE.md). Принять сознательно как отдельное расширение, не тихо.
- **Где:** поверх П3 + оверлея шага 3 (шаблон = сохранённый `FitResult`/контур; матчинг по
  близости длины/ширины контуров). Усилия средние-высокие. Связано с каталогом (Шаг 6).
- **UI — двухпанельное сравнение:** слева сохранённый рентген эталонной (удобной) обуви,
  справа пользователь загружает модель-кандидата → два оверлея в одном масштабе рядом,
  видно где кандидат у́же/просторнее эталона. Переиспользует движок оверлея дважды.
- **UX — профиль стопы один раз:** замер (длина/ширина/A4 и доп.опции) вводится **однократно**
  и переиспользуется во всех сравнениях; не просить заново каждый раз.
  **⚠️ Зависимость:** требует сохранения профиля стопы (storage/аккаунт). На MVP БД нет —
  отложена на **Шаг 6**; значит «один раз и навсегда» возможно только после персистентности
  (до этого — хотя бы в рамках одной сессии держим в состоянии App, как сейчас `foot`).

## Приоритет 4 — после шага 4

### ⬜ Тип стопы по пальцам (египетская / греческая / римская + палец Мортона)
Какой палец длиннее. Закрывает дыру геометрии длина×ширина: где давит носок (греческая
стопа / Мортон + узкий/низкий toe-box = давление на 2-й палец).
- **Честность/ценность:** пользователь сам определяет тип по картинке (user-provided → проходит фильтр).
- **⚠️ Только локальная нота, НЕ меняет длину:** длина уже = пятка→самый длинный палец, так что
  пересчитывать её по типу = двойной счёт/выдумка. Единственный защитимый эффект — **нота про носок**,
  связывающая новое поле `toeType` с **уже существующим, но СЕЙЧАС НЕиспользуемым** `attrs.toeBox`
  (его первый потребитель!): напр. греческая/Мортон + toeBox low/standard → «возможно давление на
  2-й палец; узкий носок», помечено приблизительно.
- **Где:** опциональное поле `toeType` в `FootMeasurement` + нота в вердикте (шаг 4+).

## Осознанно отложено (решение зафиксировано, чтобы не передумывать молча)

### ⏸ Обхват / объём / высота подъёма / свод стопы (всё «третье измерение»)
Обхваты (ball/instep/waist girth), объём, высота подъёма, свод (planus/normal/cavus —
Arch Index/FPI/calcaneal pitch). По исследованию (§1.3, §2) это **главные факторы посадки**
(3D-морфология: первые 6 мод = 92,59% вариации; мода №1 — высота свода).
- **Почему не сейчас:** требует данных по **внутренней высоте/обхвату колодки**, которых из
  закрытых колодок не взять (выдумывание), а клинический свод вообще требует сканера/осмотра
  (не user-provided). Это же исследование — честное **обоснование оставаться в 2D** (дискретные
  ширина-категория/тип пальцев + 2D — осознанное приближение), а не повод добавлять моды.
- **Вернуться** можно только через 3D-скан или якорь «удобная обувь» (П3) — там обхват/объём
  приходят с реальной обуви.

### ⏸ Heel-to-ball / выравнивание самой широкой части (arch length)
Brannock-стандарт (§2): самый широкий сустав стопы должен совпасть с самым широким местом колодки,
даже если общая длина меньше. Сейчас ширину сравниваем «вообще», без продольной позиции.
- **Почему не сейчас:** сторона стопы **измерима по A4** (зависит от шага 2.6), но позиция самого
  широкого места **колодки** — закрытые данные. Записываем как известный неучтённый фактор; позже —
  максимум приблизительный **флаг** «самая широкая часть может не совпасть», НИКОГДА не точный вердикт.

### ⏸ Аллометрия ширины (вместо плоского `WIDTH_RATIO = 0.40`)
По §2 обхват растёт непропорционально длине (b = 0.743 муж. / 0.866 жен., < 1) → плоский ratio
переширяет большие стопы и недоширяет малые.
- **Почему не сейчас:** трогает уже отгруженную модель шага 3, а выигрыш **только направленный**
  (это популяционный обхват, не ширина колодки). Сделать позже как ревизию шага 3 с тестами на
  **монотонность** (малая стопа → относительно шире, большая → у́же), помечено приблизительным.

### ⏸ Полнота A–EEEE (точнее, чем 3 категории) — только ввод пользователя
Полнота — независимая ось (§2). Дать опциональный выбор полноты (A–EEEE / medium-wide-extra),
маппящийся в более тонкий `FOOT_WIDTH_RATIO`, **когда ширина в мм не задана**.
- **Почему отложено:** только как **пользовательский** ввод (полноту не выводим!), измеренная
  ширина всегда приоритетнее; добавлять лишь если не усложнит основной путь.

### ⏸ Подкладка / перфорация как модификаторы смягчения
Эффект реален (§3.1: подкладка −14…−32% жёсткости, перфорация ослабляет), но достоверных данных
по конкретной модели нет (колодки закрыты, LLM выдумает). Отложено до надёжного источника/ввода.

### ⏸ Петля обратной связи «реально подошло?»
Сбор «подошло/жало/болталось» после покупки → калибровка констант (`TOE_ROOM`, пороги зон,
сдвиги репутации) на реальных данных. В долгую превращает приближения в данные. Отложено.

### ⏸ Бренд-уровневые тенденции колодок
«Бренд N в среднем узит». Данные мягкие, и per-model `widthReputation` это уже покрывает —
отдельная ценность низкая.
